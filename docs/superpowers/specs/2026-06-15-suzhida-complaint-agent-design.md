# “诉智达”投诉智能处置与闭环管理系统需求设计

## 1. 文档目的

本文定义“诉智达”MVP 的业务范围、系统架构、Agent 工作流、知识库方案、接口边界、数据模型、评估体系和实施阶段。

项目同时服务于两个目标：

1. 建成一个可以实际运行和演示的投诉智能处置 MVP。
2. 形成一个业务动机清晰、技术取舍可信、能够经受面试追问的企业级 AI 应用项目。

本文只描述需求与设计，不包含实现代码和详细开发任务。

## 2. 业务背景

客服团队每日需要处理大量投诉。一线坐席对复杂业务规则、历史工单和属地政策的掌握程度不一致，疑难投诉依赖少数专家，容易出现以下问题：

- 投诉分类和关键信息提取依赖人工经验。
- 查询业务规则和相似工单耗时较长。
- 不同坐席给出的处置方案标准不一致。
- 方案缺乏明确依据，质检和复盘困难。
- 复杂投诉处理周期长，重复投诉和升级投诉风险较高。

“诉智达”作为现有客服系统的辅助能力，不替代人工坐席，也不直接自动关闭投诉。系统根据投诉内容检索相关规则和历史案例，生成有依据的处置建议，并由坐席进行最终确认。

## 3. 产品目标与非目标

### 3.1 产品目标

- 帮助坐席快速理解投诉意图、情绪和关键业务实体。
- 从业务规则、历史工单和处置模板中检索可靠依据。
- 生成结构化、可执行、可追溯的投诉处置建议。
- 在低置信度、高风险或规则冲突场景中主动转人工。
- 记录人工采纳、修改和驳回结果，形成效果评估闭环。
- 同时提供可演示的 Web 工作台和可集成的后端 API。

### 3.2 MVP 非目标

- 不自动派单或自动关闭投诉。
- 不直接回写真实生产客服系统。
- 不构建开放式多 Agent 协作平台。
- 不在第一阶段实施大规模模型微调。
- 不在第一阶段建设完整运营管理后台。
- 不在第一阶段完成生产级 Kubernetes 集群部署。

## 4. 用户角色

### 4.1 客服坐席

录入投诉内容，查看意图、实体、检索证据和处置建议，对建议执行采纳、修改或驳回。

### 4.2 质检与管理人员

查看方案质量、人工采纳情况、异常案例及基础效果指标，为规则和流程优化提供依据。

### 4.3 知识维护人员

维护脱敏历史工单、业务规则、FAQ 和标准处置模板，执行知识入库和版本更新。

## 5. 核心业务流程

```text
投诉录入
  -> 意图识别与实体抽取
  -> 查询改写
  -> 混合检索
  -> 证据重排序
  -> 处置方案生成
  -> 规则与依据校验
  -> 人工确认、修改或驳回
  -> 结果归档与反馈
```

系统生成的内容始终属于“辅助建议”。涉及高风险、低置信度、敏感投诉、依据不足或规则冲突的场景，必须明确提示人工处理。

## 6. 总体架构选择

### 6.1 选择结论

系统采用“单 Agent + LangGraph 状态机 + 局部循环”的架构。

不采用简单线性 RAG，是因为系统需要根据检索和校验结果执行重试、降级或转人工；不采用完整 Multi-Agent，是因为投诉处置属于边界明确的强流程业务，多 Agent 会增加推理成本、延迟、上下文漂移和线上调试难度。

### 6.2 系统组件

```text
客服 Web 工作台
  -> 后端 API / SSE 网关
  -> LangGraph 投诉处置工作流
       -> 意图与实体识别
       -> 查询改写
       -> 检索编排
       -> 方案生成
       -> 规则校验
       -> 人工确认
  -> 模型网关
       -> vLLM 大模型服务
       -> Embedding 服务
       -> Rerank 服务
  -> 数据服务
       -> Milvus
       -> Elasticsearch
       -> Redis
       -> MySQL
```

### 6.3 组件边界

- Web 工作台：只负责展示、录入、流式交互和人工反馈，不包含 Agent 决策逻辑。
- API 网关：负责鉴权预留、请求校验、限流、SSE 连接和错误响应。
- LangGraph 工作流：负责状态流转、工具选择、条件分支、循环和人工中断。
- 模型网关：屏蔽具体模型服务差异，统一超时、重试、降级和调用审计。
- 检索服务：统一封装 Milvus、Elasticsearch、Redis 的召回和融合逻辑。
- 校验服务：执行确定性规则、引用检查、风险识别和输出格式校验。
- 数据服务：负责会话、证据、方案、反馈、评估和知识元数据持久化。

## 7. LangGraph 工作流

### 7.1 状态定义

工作流状态至少包含：

- `session_id`
- `complaint_text`
- `intent`
- `emotion`
- `entities`
- `normalized_query`
- `retrieved_evidence`
- `generated_solution`
- `validation_result`
- `risk_level`
- `retry_count`
- `human_action`
- `error`

### 7.2 节点设计

#### intent_node

识别投诉类型、情绪、用户诉求以及地州、产品、套餐、账期等关键实体。输出必须符合结构化 Schema。

#### query_rewrite_node

将口语化、方言化或信息混杂的投诉描述转换为适合规则和工单检索的查询。保留原始诉求，不得生成用户未表达的事实。

#### retrieve_node

调用 Milvus 向量检索、Elasticsearch BM25 检索和 Redis 高频模板查询，合并候选证据。

#### rerank_node

使用 BGE-reranker 对候选结果重排序，输出 Top-K 证据及分数。

#### generate_node

基于投诉内容、结构化实体和检索证据生成处置方案。方案必须包含问题判断、建议步骤、风险提示和引用依据。

#### validate_node

检查：

- 是否存在有效引用。
- 引用编号是否能在召回证据中找到。
- 方案结构是否完整。
- 是否包含无依据承诺或敏感表述。
- 检索置信度和生成结果是否满足阈值。
- 是否出现规则冲突或高风险投诉。

#### human_review_node

暂停自动流程，等待坐席采纳、修改或驳回。高风险场景必须显示人工优先标识。

#### feedback_node

记录人工操作、修改内容、驳回原因、最终方案和节点运行信息，为离线评估及后续优化提供数据。

### 7.3 分支与循环

```text
校验通过
  -> 人工确认

依据不足且 retry_count 未超限
  -> 查询改写
  -> 重新检索
  -> 重新生成

输出格式问题且 retry_count 未超限
  -> 重新生成

高风险、规则冲突、持续失败或达到重试上限
  -> 人工优先
```

单次会话的自动重试次数默认不超过 2 次，避免无限循环和不可控的推理成本。

## 8. 知识库与 RAG

### 8.1 MVP 数据范围

- 200 至 500 条脱敏历史投诉工单。
- 30 至 80 条业务规则、资费政策和服务规范。
- 20 至 50 条高频投诉处置模板。
- 少量云南本地方言、口语表达和标准业务术语对照样例。

### 8.2 知识分类

- 历史工单库：投诉描述、原因分析、处理过程和最终结果。
- 业务规则库：套餐、退费、升级处理、服务 SLA 等条款。
- 处置模板库：高频投诉的标准步骤、回复框架和风险提示。

### 8.3 入库处理

- 历史工单按业务字段结构化，敏感信息在入库前脱敏。
- 规则文档按章节和条款分块，保留条款编号和版本。
- Chunk 建议为 300 至 600 token，重叠 50 至 80 token。
- Metadata 至少包含来源类型、来源编号、业务类型、地州、产品、时间、版本和条款编号。
- Embedding 使用 BGE-M3，向量写入 Milvus。
- 原文和可检索字段写入 Elasticsearch。
- 高频模板和热点查询结果可缓存在 Redis。

### 8.4 检索流程

```text
规范查询
  -> Milvus 语义召回
  -> Elasticsearch BM25 召回
  -> 候选去重与分数归一化
  -> 融合排序
  -> BGE-reranker 重排序
  -> Top-K 证据
```

MVP 默认向生成节点提供 Top-3 证据，同时保留 Top-5 结果用于界面展示和评估。融合权重和阈值通过离线评估集调优，不在需求阶段固定为不可变常量。

### 8.5 知识更新

每次导入需要记录数据批次、来源、版本、处理状态和失败原因。新版本规则生效后，旧版本应保留审计信息但不参与默认检索。

## 9. 模型与私有化部署

### 9.1 模型服务

- 主模型：Qwen2.5-14B 或同级中文开源模型。
- 推理框架：vLLM。
- Embedding：BGE-M3。
- Rerank：BGE-reranker。

模型通过内部 HTTP 服务提供能力，业务后端不得直接依赖模型进程内部实现。

### 9.2 生产部署挑战

#### 显存与吞吐

Qwen2.5-14B 可使用 INT4 或 INT8 量化降低显存占用。需要结合实际 GPU、上下文长度、`max_num_seqs`、KV Cache 和 continuous batching 做压测。简历中的生产指标属于目标和压测结果，不应仅根据理论值推断。

#### 首字延迟

后端采用 SSE 流式输出。意图识别和检索阶段先返回阶段状态，生成节点启动后返回模型 token。模型预热、Prompt 长度、并发队列和检索耗时需要分别监控。

#### 高可用与降级

- 模型请求超时后可重试一次。
- 主模型不可用时返回检索证据和人工处理提示，不生成无依据方案。
- Milvus 不可用时可降级为 Elasticsearch 检索。
- Elasticsearch 不可用时可使用 Milvus 单路召回，但必须标记降级状态。
- 两种检索服务均不可用时直接转人工。

#### 隐私与审计

模型、检索和数据服务部署在内网。手机号、身份证、地址等敏感信息在日志、评估集和训练集使用前必须脱敏。模型调用需记录请求 ID、模型版本、耗时、状态和 token 使用量，但不得记录未脱敏原文。

## 10. Web 工作台

### 10.1 页面布局

- 左侧：新建投诉、历史会话和处理状态。
- 中间：投诉输入、工作流进度和流式处置建议。
- 右侧：意图、情绪、实体、检索证据、校验结果和风险提示。
- 底部操作区：采纳、编辑后采纳、驳回及反馈原因。

### 10.2 核心交互

1. 坐席输入投诉内容并发起分析。
2. 页面展示工作流当前阶段。
3. 系统展示结构化识别结果和检索依据。
4. 处置建议以流式方式展示。
5. 坐席查看风险和校验结果。
6. 坐席采纳、修改或驳回。
7. 页面展示归档成功或错误兜底信息。

## 11. API 需求

### 11.1 核心接口

```text
POST /api/v1/complaints/sessions
POST /api/v1/complaints/{session_id}/messages
GET  /api/v1/complaints/{session_id}
POST /api/v1/complaints/{session_id}/feedback
GET  /api/v1/knowledge/search
GET  /api/v1/metrics/summary
```

`messages` 接口通过 SSE 返回阶段事件和生成内容。事件类型至少包括：

- `workflow_started`
- `intent_completed`
- `retrieval_completed`
- `generation_delta`
- `validation_completed`
- `human_review_required`
- `workflow_completed`
- `workflow_failed`

### 11.2 幂等与错误处理

- 创建会话支持客户端请求 ID，防止重复创建。
- 同一会话同一时刻只允许一个工作流运行。
- 人工反馈接口必须具备幂等性。
- API 返回统一错误码、可读消息和追踪 ID。
- 客户端断开 SSE 不应导致已完成的工作流状态丢失。

## 13. 安全与人机协同

- 最终处置权归人工坐席。
- 低置信度、高风险、规则冲突和检索失败必须转人工。
- 生成内容不得承诺超出规则授权的退费、补偿或处理时限。
- 所有生成方案必须包含可追溯依据。
- 原始投诉数据、日志、评估数据和知识库数据执行分级权限控制。
- MVP 可以简化用户体系，但接口和数据模型需保留角色边界。

## 14. 可观测性

### 14.1 Monitoring

- API 请求量、错误率和延迟。
- SSE 连接数和异常断开数。
- 各 LangGraph 节点耗时、成功率和重试次数。
- Milvus、Elasticsearch、Redis 和 MySQL 健康状态。
- 模型队列长度、首 token 延迟、生成速度和 GPU 使用率。

### 14.2 Tracing

每个投诉会话生成统一追踪 ID，串联 API、工作流节点、检索、模型调用和数据库写入。追踪数据不得包含未脱敏的投诉原文。

### 14.3 Evaluation

记录模型版本、Prompt 版本、知识版本和工作流版本，保证效果变化可回溯。

## 15. 验收指标

### 15.1 业务指标

- 人工采纳率。
- 平均处置时长。
- 上线后长期观察重复投诉率。

### 15.2 模型与检索指标

- 意图识别准确率。
- Milvus + Elasticsearch 混合检索 Recall@5。
- rerank 后 Top-3 证据命中率。
- 方案依据引用完整率。
- 无依据表述比例。

### 15.3 工程指标

- P99 首字延迟目标小于 2 秒。
- MVP 压测目标为平峰 50 路并发。
- 生产架构支持通过模型副本和网关水平扩展至高峰 200 路。
- 核心 API 成功率目标不低于 99%。
- 高风险和持续失败场景必须转人工，不允许静默失败。

以上数值均需通过明确的测试数据集、硬件环境和压测脚本验证，并在报告中注明测试条件。

## 16. 测试策略

### 16.1 单元测试

覆盖结构化输出解析、检索融合、阈值分支、校验规则、幂等处理和数据持久化。

### 16.2 工作流测试

使用固定投诉样例验证节点状态、条件边、循环上限、人工中断、恢复和异常降级。

### 16.3 离线评估

准备 100 至 200 条标注投诉样例，覆盖高频、长尾、方言、信息缺失、规则冲突和敏感投诉。

### 16.4 人工评估

由业务人员按“可直接采纳、修改后采纳、不可用”标注方案，并记录原因。

### 16.5 性能测试

分别测试 API、检索、rerank、模型首 token 延迟和端到端耗时，避免只报告整体平均值。

## 17. 实施阶段

### 阶段一：MVP 核心闭环

- 简化客服工作台。
- 投诉会话 API 和 SSE。
- LangGraph 单 Agent 状态机。
- Milvus + Elasticsearch 混合检索。
- BGE-reranker 重排序。
- Redis 高频模板。
- MySQL 会话、证据、方案和反馈存储。
- 规则校验与人工确认。
- 基础离线评估与链路日志。

### 阶段二：生产增强

- 扩充脱敏知识数据。
- 建立知识导入、版本和回滚机制。
- 完善权限、审计和脱敏。
- 增加 Prometheus + Grafana。
- 优化 vLLM 量化、并发和扩缩容。
- 小范围坐席灰度测试。

### 阶段三：效果优化

- 根据真实反馈优化 Prompt、融合权重和 rerank 策略。
- 增加方言和属地规则适配。
- 在数据质量和收益得到验证后评估 LoRA 微调。
- 建立人工反馈、回归评估和模型迭代闭环。

## 18. 团队开发规范

- 使用 `main`、`develop`、`feature/*` 三级分支体系。
- 功能开发从最新 `develop` 创建独立分支。
- 提交信息遵循 Conventional Commits，例如 `feat`、`fix`、`docs`、`refactor`、`test`。
- 合并必须通过 Pull Request、代码审查和自动化测试。
- 冲突在功能分支解决并完成本地回归后再提交。
- 模型、Prompt、知识库和工作流变更均需记录版本及评估结果。
- 每次发布遵循需求确认、方案评审、开发、自测、评审、灰度、上线和复盘流程。

## 19. MVP 完成标准

满足以下条件后，MVP 才可视为完成：

- 可通过 Web 页面创建投诉会话并获得流式建议。
- LangGraph 核心节点、条件分支和有限循环正常工作。
- Milvus 和 Elasticsearch 均参与混合检索。
- 每个生成方案可展示真实存在的引用证据。
- 校验失败、服务异常和低置信度场景有明确兜底。
- 坐席可以采纳、编辑或驳回，反馈可持久化。
- 离线评估、链路测试和基础性能测试可重复执行。
