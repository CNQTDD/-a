# “诉智达”投诉智能处置与闭环管理系统需求设计

## 1. 文档目标

本文档定义“诉智达”MVP 的业务范围、系统架构、智能体工作流、知识库方案、接口边界、数据模型、评估体系与实施阶段。

项目同时服务两个目标：

1. 交付一个可真实运行、可演示、可本地部署的投诉智能处置 MVP。
2. 形成一个业务动机清晰、技术取舍合理、可经受面试追问的企业级 AI 应用项目方案。

本文只描述需求与设计，不包含实现代码与逐任务开发步骤。

## 2. 业务背景

客服团队每天需要处理大量投诉。一线坐席对复杂业务规则、历史工单和属地政策的掌握程度并不一致，疑难投诉往往依赖少数专家，常见问题包括：

- 投诉分类和关键信息抽取依赖人工经验。
- 查询业务规则和相似工单耗时较长。
- 不同坐席给出的处置方案标准不一致。
- 方案缺少明确依据，质检和复盘困难。
- 复杂投诉处理周期长，重复投诉和升级投诉风险较高。

“诉智达”作为现有客服系统的辅助能力，不替代人工坐席，也不直接自动结案。系统根据投诉内容检索规则与历史案例，生成有依据的处置建议，再由坐席做最终确认。

## 3. 产品目标与非目标

### 3.1 产品目标

- 帮助坐席快速理解投诉意图、情绪和关键业务实体。
- 从业务规则、历史工单和处置模板中检索可靠依据。
- 生成结构化、可执行、可追溯的投诉处置建议。
- 在低置信度、高风险、规则冲突等场景下主动转人工。
- 记录人工采纳、修改和驳回结果，形成效果评估闭环。
- 同时提供可演示的网页工作台和可集成的后端 API。

### 3.2 MVP 非目标

- 不自动派单或自动关闭投诉。
- 不直接回写真实生产客服系统。
- 不构建开放式多智能体协作平台。
- 不在第一阶段实施大规模模型微调。
- 不在第一阶段建设完整运营管理后台。
- 不在第一阶段完成生产级 `Kubernetes` 集群部署。

## 4. 用户角色

### 4.1 客服坐席

录入投诉内容，查看意图、实体、检索证据与处置建议，并对建议执行采纳、修改或驳回。

### 4.2 质检与管理人员

查看方案质量、人工采纳情况、异常案例与基础效果指标，为规则和流程优化提供依据。

### 4.3 知识维护人员

维护脱敏历史工单、业务规则、FAQ 和标准处置模板，执行知识入库与版本更新。

## 5. 核心业务流程

```text
投诉录入
  -> 意图识别与实体抽取
  -> 查询改写
  -> 混合检索
  -> 证据重排
  -> 处置方案生成
  -> 规则与依据校验
  -> 人工确认、修改或驳回
  -> 结果归档与反馈
```

系统生成的内容始终属于“辅助建议”。凡是高风险、低置信度、敏感投诉、依据不足或规则冲突场景，都必须明确提示人工处理。

## 6. 总体架构选择

### 6.1 架构结论

系统采用“单智能体 + LangGraph 状态机 + 局部循环”的总体架构。

- 不采用简单线性 RAG，是因为系统需要根据检索和校验结果执行重试、降级或转人工。
- 不采用完整多智能体，是因为投诉处置属于边界清晰的强流程业务，多 Agent 会显著增加推理成本、延迟、上下文漂移和线上调试难度。

### 6.2 系统组件

```text
客服网页工作台
  -> 后端 API / SSE 网关
  -> LangGraph 投诉处置工作流
       -> 意图与实体识别
       -> 查询改写
       -> 检索编排
       -> 方案生成
       -> 规则校验
       -> 人工复核
  -> 模型网关
       -> vLLM 大模型服务
       -> 嵌入服务
       -> 重排服务
  -> 数据服务
       -> Milvus
       -> Elasticsearch
       -> Redis
       -> MySQL
```

### 6.3 组件边界

- 网页工作台只负责展示、录入、流式交互和人工反馈，不承载智能体决策逻辑。
- API 网关负责鉴权预留、请求校验、限流、SSE 连接与统一错误响应。
- LangGraph 工作流负责状态流转、条件分支、局部循环和人工中断。
- 模型网关负责统一超时、重试、降级、审计与服务隔离，不让业务代码直接耦合底层模型实现。
- 检索服务统一封装 `Milvus`、`Elasticsearch`、`Redis` 的召回与融合逻辑。
- 校验服务执行确定性规则、引用检查、风险识别与输出格式校验。
- 数据服务负责会话、证据、方案、反馈、评估和知识元数据持久化。

## 7. LangGraph 工作流设计

### 7.1 状态字段

工作流状态至少包含：

- `session_id`
- `complaint_text`
- `intent`
- `emotion`
- `entities`
- `normalized_query`
- `retrieved_evidence`
- `generated_solution`
- `validation_result`
- `risk_level`
- `retry_count`
- `human_action`
- `error`

### 7.2 节点设计

#### 意图识别节点（`intent_node`）

识别投诉类型、情绪、用户诉求以及地州、产品、套餐、账期等关键实体，输出必须符合结构化 Schema。

#### 查询改写节点（`query_rewrite_node`）

将口语化、方言化或信息混杂的投诉描述转换为适合规则与工单检索的查询表达，保留原始诉求，不得补造用户未明确表达的事实。

#### 检索节点（`retrieve_node`）

调用 `Milvus` 向量检索、`Elasticsearch BM25` 检索和 `Redis` 高频模板查询，合并候选证据。

#### 重排节点（`rerank_node`）

使用 `BGE-reranker` 对候选结果重排序，输出前 `K` 条证据及得分。

#### 生成节点（`generate_node`）

基于投诉内容、结构化实体和检索证据生成处置方案。方案必须包含问题判断、建议步骤、风险提示与引用依据。

#### 校验节点（`validate_node`）

检查以下内容：

- 是否存在有效引用。
- 引用编号能否在召回证据中找到。
- 方案结构是否完整。
- 是否包含无依据承诺或敏感表述。
- 检索置信度和生成结果是否满足阈值。
- 是否存在规则冲突或高风险投诉。

#### 人工复核节点（`human_review_node`）

暂停自动流程，等待坐席采纳、修改或驳回。高风险场景必须醒目显示人工优先标识。

#### 反馈节点（`feedback_node`）

记录人工动作、修改内容、驳回原因、最终方案和节点运行信息，用于离线评估与持续优化。

### 7.3 分支与循环

```text
校验通过
  -> 人工确认

依据不足 且 retry_count 未超过上限
  -> 查询改写
  -> 重新检索
  -> 重新生成

输出格式问题 且 retry_count 未超过上限
  -> 重新生成

高风险、规则冲突、持续失败或达到重试上限
  -> 人工优先
```

单次会话默认自动重试次数不超过 `2` 次，避免无限循环和不可控推理成本。

## 8. 知识库与 RAG

### 8.1 MVP 数据范围

- `200` 到 `500` 条脱敏历史投诉工单。
- `30` 到 `80` 条业务规则、资费政策和服务规范。
- `20` 到 `50` 条高频投诉处置模板。
- 少量云南本地方言、口语表达和标准业务术语对照样例。

### 8.2 知识分类

- 历史工单库：投诉描述、原因分析、处理过程与最终结果。
- 业务规则库：套餐、退费、升级处理、服务 `SLA` 等条款。
- 处置模板库：高频投诉的标准步骤、回复框架与风险提示。

### 8.3 入库处理

- 历史工单按业务字段结构化，敏感信息在入库前脱敏。
- 规则文档按章节和条款分块，保留条款编号和版本信息。
- 分块建议为 `300` 到 `600 token`，重叠 `50` 到 `80 token`。
- 元数据至少包含来源类型、来源编号、业务类型、地州、产品、时间、版本和条款编号。
- 向量嵌入使用 `BGE-M3`，写入 `Milvus`。
- 原文和可检索字段写入 `Elasticsearch`。
- 高频模板和热点查询结果可缓存至 `Redis`。

### 8.4 检索流程

```text
规范查询
  -> Milvus 语义召回
  -> Elasticsearch BM25 召回
  -> 候选去重与分数归一化
  -> 融合排序
  -> BGE-reranker 重排
  -> 前 K 证据
```

MVP 默认向生成节点提供前 `3` 条证据，同时保留前 `5` 条结果用于界面展示和评估。融合权重和阈值应通过离线评估调优，不在需求阶段写死。

### 8.5 知识更新

每次导入都要记录数据批次、来源、版本、处理状态和失败原因。新规则生效后，旧版本保留审计信息，但不应继续参与默认检索。

## 9. 模型与私有化部署

### 9.1 模型服务

- 主模型：`Qwen2.5-14B` 或同级中文开源模型。
- 推理框架：`vLLM`。
- 向量嵌入：`BGE-M3`。
- 重排：`BGE-reranker`。

模型必须通过内部 `HTTP` 服务提供能力，业务后端不得直接依赖模型进程内部实现。

### 9.2 部署挑战

#### 显存与吞吐

`Qwen2.5-14B` 可使用 `INT4` 或 `INT8` 量化降低显存占用，但实际容量仍需结合 `GPU`、上下文长度、`max_num_seqs`、`KV Cache` 与 `continuous batching` 压测验证。简历中出现的并发指标必须来自实测，而不是纯理论估算。

#### 首字延迟

后端采用 `SSE` 流式输出。意图识别和检索阶段先返回阶段状态，生成节点启动后再持续返回模型输出。模型预热、Prompt 长度、并发队列与检索耗时都需要单独监控。

#### 高可用与降级

- 模型请求超时后可重试一次。
- 主模型不可用时，只返回检索证据和人工处理提示，不生成无依据方案。
- `Milvus` 不可用时可降级为 `Elasticsearch` 单路检索。
- `Elasticsearch` 不可用时可降级为 `Milvus` 单路检索，但必须标记降级状态。
- 两类检索服务同时不可用时直接转人工。

#### 隐私与审计

模型、检索和数据服务都部署在内网。手机号、身份证、地址等敏感信息在日志、评估集和训练集使用前必须脱敏。模型调用应记录请求 `ID`、模型版本、耗时、状态和 `token` 用量，但不得记录未脱敏原文。

## 10. 网页工作台

### 10.1 页面布局

- 左侧：新建投诉、历史会话与处理状态。
- 中间：投诉输入、工作流进度与流式处置建议。
- 右侧：意图、情绪、实体、检索证据、校验结果和风险提示。
- 底部：采纳、修改后采纳、驳回与反馈原因。

### 10.2 核心交互

1. 坐席输入投诉内容并发起分析。
2. 页面显示当前工作流阶段。
3. 系统展示结构化识别结果和检索依据。
4. 处置建议以流式形式逐步呈现。
5. 坐席查看风险与校验结果。
6. 坐席采纳、修改或驳回。
7. 页面显示归档结果或兜底错误提示。

## 11. API 需求

### 11.1 核心接口

```text
POST /api/v1/complaints/sessions
POST /api/v1/complaints/{session_id}/messages
GET  /api/v1/complaints/{session_id}/events
GET  /api/v1/complaints/{session_id}
POST /api/v1/complaints/{session_id}/feedback
GET  /api/v1/knowledge/search
GET  /api/v1/metrics/summary
```

`messages` 接口负责原子性启动工作流并返回 `run_id`；`events` 接口通过 `SSE` 返回可重放的阶段事件与生成内容，并支持 `Last-Event-ID` 重连。工作流执行与客户端连接必须解耦，前端断开连接不应终止后台处理。

事件类型至少包括：

- `workflow_started`
- `intent_completed`
- `retrieval_completed`
- `generation_delta`
- `validation_completed`
- `human_review_required`
- `workflow_completed`
- `workflow_failed`

### 11.2 幂等与错误处理

- 创建会话支持客户端请求 ID，避免重复创建。
- 同一会话同一时刻只允许一个工作流运行。
- 人工反馈接口必须具备幂等性。
- API 返回统一错误结构，包含错误码、可读消息与追踪 ID。
- 客户端断开 `SSE` 不应导致已完成工作流状态丢失。

## 13. 安全与人机协同

- 最终处置权归人工坐席。
- 低置信度、高风险、规则冲突和检索失败场景必须转人工。
- 生成内容不得承诺超出规则授权的退费、补偿或处理时限。
- 所有生成方案必须带可追溯依据。
- 原始投诉数据、日志、评估数据与知识库数据要执行分级权限控制。
- MVP 可以简化用户体系，但接口和数据模型必须保留角色边界。

## 14. 可观测性

### 14.1 监控

- API 请求量、错误率和延迟。
- `SSE` 连接数与异常断开数。
- LangGraph 各节点耗时、成功率和重试次数。
- `Milvus`、`Elasticsearch`、`Redis`、`MySQL` 健康状态。
- 模型队列长度、首 `token` 延迟、生成速率和 `GPU` 使用率。

### 14.2 链路追踪

每个投诉会话生成统一追踪 ID，串联 API、工作流节点、检索、模型调用和数据库写入。追踪数据不得包含未脱敏投诉原文。

### 14.3 评估

必须记录模型版本、Prompt 版本、知识版本与工作流版本，确保效果变化可回溯。

## 15. 验收指标

### 15.1 业务指标

- 人工采纳率。
- 平均处置时长。
- 上线后持续观察重复投诉率。

### 15.2 模型与检索指标

- 意图识别准确率。
- `Milvus + Elasticsearch` 混合检索 `Recall@5`。
- 重排后前 `3` 证据命中率。
- 方案引用完整率。
- 无依据表述占比。

### 15.3 工程指标

- `P99` 首字延迟目标小于 `2` 秒。
- MVP 压测目标为平峰 `50` 路并发。
- 生产架构支持通过模型副本和网关水平扩展到高峰 `200` 路并发。
- 核心 API 成功率目标不低于 `99%`。
- 高风险与持续失败场景必须转人工，不允许静默失败。

所有数值都必须通过明确的数据集、硬件环境与压测脚本验证，并在报告中注明测试条件。

## 16. 测试策略

### 16.1 单元测试

覆盖结构化输出解析、检索融合、阈值分支、校验规则、幂等处理与数据持久化。

### 16.2 工作流测试

使用固定投诉样例验证节点状态、条件边、循环上限、人工中断、恢复与异常降级。

### 16.3 离线评估

准备 `100` 到 `200` 条标注投诉样例，覆盖高频、长尾、方言、信息缺失、规则冲突与敏感投诉。

### 16.4 人工评估

由业务人员按“可直接采纳、修改后采纳、不可用”对方案打标，并记录原因。

### 16.5 性能测试

分别测试 API、检索、重排、模型首 `token` 延迟与端到端耗时，避免只报告整体平均值。

## 17. 实施阶段

### 阶段一：MVP 核心闭环

- 简化客服工作台。
- 投诉会话 API 与 `SSE`。
- LangGraph 单智能体状态机。
- `Milvus + Elasticsearch` 混合检索。
- `BGE-reranker` 重排。
- `Redis` 高频模板。
- `MySQL` 会话、证据、方案与反馈存储。
- 规则校验与人工确认。
- 基础离线评估与链路日志。

### 阶段二：生产增强

- 扩充脱敏知识数据。
- 建立知识导入、版本和回滚机制。
- 完善权限、审计与脱敏。
- 增加 `Prometheus + Grafana`。
- 优化 `vLLM` 量化、并发与扩缩容。
- 小范围坐席灰度测试。

### 阶段三：效果优化

- 根据真实反馈优化 Prompt、融合权重和重排策略。
- 增加方言和属地规则适配。
- 在数据质量和收益得到验证后评估 `LoRA` 微调。
- 建立人工反馈、回归评估和模型迭代闭环。

## 18. 团队开发规范

- 使用 `main`、`develop`、`feature/*` 的三级分支体系。
- 功能开发从最新 `develop` 创建独立分支。
- 提交信息遵循 Conventional Commits，例如 `feat`、`fix`、`docs`、`refactor`、`test`。
- 合并必须经过 Pull Request、代码审查和自动化测试。
- 冲突在功能分支解决并完成本地回归后再提交。
- 模型、Prompt、知识库与工作流变更都要记录版本和评估结果。
- 每次发布遵循需求确认、方案评审、开发、自测、评审、灰度、上线和复盘流程。

## 19. MVP 完成标准

满足以下条件后，MVP 才可视为完成：

- 可通过网页界面创建投诉会话并获得流式建议。
- LangGraph 核心节点、条件分支和有限循环工作正常。
- `Milvus` 与 `Elasticsearch` 均参与混合检索。
- 每个生成方案都能展示真实存在的引用证据。
- 校验失败、服务异常和低置信度场景都有明确兜底。
- 坐席可以采纳、编辑或驳回，反馈可持久化。
- 离线评估、链路测试和基础性能测试可重复执行。
- 文档中记录测试数据、模型版本、硬件环境与指标结果。
